## API Usage Examples

### Check Status (Public)
```bash
curl http://localhost:8080/api/killswitch/status
```

### Activate (Protected)
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "level": "EMERGENCY_STOP",
    "reason": "Market conditions require immediate halt",
    "activatedBy": "operator-name"
  }'
```

### Deactivate (Protected)
```bash
curl -X POST http://localhost:8080/api/killswitch/deactivate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "deactivatedBy": "operator-name",
    "reason": "Issue resolved"
  }'
```

## Performance Characteristics

- **Status Query**: 5-second cache (minimal database load)
- **Activation**: Single database write + optional order cancellation (sequential)
- **Order Cancellation**: One API call per open order (iterative, not parallelized)
- **Worker Check**: In-memory cache hit (no database query under normal conditions)

## Security Features

1. Secret validation for all write operations
2. Audit trail with operator name and IP address
3. Failed attempts logged with details
4. Public read-only status endpoint (no sensitive data)
5. State persists across restarts (PostgreSQL)

## Lessons Learned

1. **Test Strategy**: Integration tests requiring database should be skipped by default; unit tests with fakes preferred
2. **Caching**: 5-second cache significantly reduces database load for status queries
3. **Worker Design**: Continue health checks even when paused (important for monitoring)
4. **Audit Trail**: IP address logging valuable for security investigation
5. **Documentation**: Comprehensive operations guide essential for emergency procedures
