# M7.2 Implementation Plan: Kill Switch

## Overview
Implement a three-level kill switch for emergency trading halt with PostgreSQL-backed state, API control, and worker integration.

## Kill Switch Levels
1. **PAUSE_NEW**: Stop accepting new alerts only (alerts worker stops)
2. **PAUSE_ALL**: Pause all workers, keep API running for status queries
3. **EMERGENCY_STOP**: Cancel all open orders + pause everything

## Implementation Phases

### Phase 1: Database Schema (20 minutes)
**File**: `scripts/db/init.sql`

Add two tables:
- `system_state`: Key-value store for kill switch and other global flags
  - Primary key: `key` (text)
  - `value` (jsonb): Contains `{active, level, reason, activated_at}`
  - `updated_at_utc`, `updated_by`
  
- `kill_switch_audit`: Audit trail for all kill switch actions
  - `audit_id` (uuid)
  - `action` (ACTIVATED/DEACTIVATED)
  - `level` (PAUSE_NEW/PAUSE_ALL/EMERGENCY_STOP)
  - `reason`, `activated_by`, `timestamp_utc`

Initialize with inactive kill switch state.

### Phase 2: Service Layer (1 hour)
**Files**: 
- `src/Mvp.Trading.Execution/IKillSwitchService.cs` (interface)
- `src/Mvp.Trading.Execution/KillSwitchService.cs` (implementation)
- `src/Mvp.Trading.Execution/KillSwitchLevel.cs` (enum)
- `src/Mvp.Trading.Execution/KillSwitchStatus.cs` (record)

**Key Methods**:
- `IsActiveAsync()`: Quick check with 5-second cache
- `GetStatusAsync()`: Full status with level/reason/timestamp
- `ActivateAsync(level, reason, by)`: Activate with audit trail
- `DeactivateAsync(by, reason)`: Deactivate with audit trail

**Special Logic**:
- `EMERGENCY_STOP` calls `ITradingProvider.CancelAllOrdersAsync()`
- 5-second in-memory cache for status checks (performance)
- All actions logged and audited

### Phase 3: API Controller (30 minutes)
**Files**:
- `src/Mvp.Trading.Api/Controllers/KillSwitchController.cs`
- `src/Mvp.Trading.Api/Models/KillSwitchActivationRequest.cs`
- `src/Mvp.Trading.Api/Models/KillSwitchDeactivationRequest.cs`
- `src/Mvp.Trading.Api/KillSwitchApiOptions.cs`

**Endpoints**:
- `GET /api/killswitch/status`: Public status check
- `POST /api/killswitch/activate`: Activate (requires secret)
- `POST /api/killswitch/deactivate`: Deactivate (requires secret)

**Security**:
- Requests must include `KILL_SWITCH_SECRET` environment variable
- IP address logged for all attempts
- Unauthorized attempts logged as warnings

### Phase 4: Worker Integration (45 minutes)
**Files**:
- `src/Mvp.Trading.Worker/AlertWorker.cs` (check PAUSE_NEW and PAUSE_ALL)
- `src/Mvp.Trading.Worker/ReconciliationWorker.cs` (check PAUSE_ALL)
- Any future workers

**Integration Points**:
- Inject `IKillSwitchService` into all workers
- Check before processing each iteration
- Respect level-specific behavior:
  - PAUSE_NEW: Only AlertWorker stops
  - PAUSE_ALL: All workers stop
  - EMERGENCY_STOP: All workers stop (orders already cancelled by service)

**Worker Behavior**:
- Log when paused due to kill switch
- Continue health checks (stay alive)
- Resume when kill switch deactivated

### Phase 5: DI Registration (15 minutes)
**Files**:
- `src/Mvp.Trading.Api/Program.cs`
- `src/Mvp.Trading.Worker/Program.cs`

Register:

Configuration structure:

## API Usage Examples

### Check Status
```bash
curl http://localhost:8080/api/killswitch/status
```

### Activate (Emergency Stop)
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "your-secret-here",
    "level": "EMERGENCY_STOP",
    "reason": "Runaway trades detected",
    "activatedBy": "operator-name"
  }'
```

### Activate (Pause New)
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "your-secret-here",
    "level": "PAUSE_NEW",
    "reason": "Market volatility too high",
    "activatedBy": "operator-name"
  }'
```

### Deactivate
```bash
curl -X POST http://localhost:8080/api/killswitch/deactivate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "your-secret-here",
    "reason": "Issue resolved, resuming operations",
    "deactivatedBy": "operator-name"
  }'
```

## Database Queries for Operators

### Check Current Kill Switch State
```sql
SELECT * FROM system_state WHERE key = 'kill_switch';
```

### View Recent Kill Switch Actions
```sql
SELECT * FROM kill_switch_audit 
ORDER BY timestamp_utc DESC 
LIMIT 10;
```

### Count Activations by Level
```sql
SELECT level, COUNT(*) as activation_count
FROM kill_switch_audit 
WHERE action = 'ACTIVATED'
  AND timestamp_utc > NOW() - INTERVAL '30 days'
GROUP BY level;
```
