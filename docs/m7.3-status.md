### Custom Application Metrics

#### Alert Processing Metrics
- **`alerts_received_alerts_total`** (Counter)
  - Labels: `exchange`, `symbol`
  - Tracks: Total webhook alerts received
  - Location: `AlertWorker.cs`

- **`alerts_processed_alerts_total`** (Counter)
  - Labels: `outcome` (accepted, rejected_llm, error)
  - Tracks: Alert processing outcomes
  - Location: `AlertWorker.cs`

- **`alert_processing_duration_seconds`** (Histogram)
  - Tracks: Time to process each alert end-to-end
  - Location: `AlertWorker.cs`

- **`queue_depth_alerts`** (Gauge)
  - Tracks: Current Redis queue length
  - Location: `AlertWorker.DequeueAsync()`

#### Execution Metrics
- **`executions_processed_total`** (Counter)
  - Labels: `outcome` (filled, placed, error, rejected_heartbeat)
  - Tracks: Trade execution results
  - Location: `ExecutionService.cs`

- **`execution_duration_seconds`** (Histogram)
  - Labels: `stage` (total)
  - Tracks: Execution timing
  - Location: `ExecutionService.cs`

- **`orders_placed_total`** (Counter)
  - Labels: `direction` (LONG/SHORT), `order_type` (MARKET, STP, TAKE_PROFIT)
  - Tracks: Orders sent to exchange
  - Location: `ExecutionService.cs`

- **`orders_filled_total`** (Counter)
  - Labels: `symbol`, `direction`
  - Tracks: Confirmed order fills
  - Location: `ExecutionService.cs`

#### Trading Activity Metrics
- **`active_trades`** (Gauge)
  - Tracks: Current open positions count
  - Location: `TradeMonitorWorker.cs`

- **`stop_loss_triggered_total`** (Counter)
  - Labels: `symbol`
  - Tracks: Stop loss invalidations
  - Location: `TradeMonitorWorker.cs`

- **`take_profit_hit_total`** (Counter)
  - Labels: `symbol`, `target` (1, 2, 3)
  - Tracks: Take profit target hits
  - Location: Future implementation

#### Reconciliation Metrics
- **`reconciliation_discrepancies`** (Gauge)
  - Tracks: Unresolved discrepancies between internal and exchange state
  - Location: `ReconciliationWorker.cs`

#### Error Tracking
- **`errors_total`** (Counter)
  - Labels: `component`, `error_type`
  - Tracks: General application errors
  - Location: All workers

- **`api_errors_total`** (Counter)
  - Labels: `provider`, `endpoint`, `error_code`
  - Tracks: Exchange API failures
  - Location: Future implementation

### HTTP/System Metrics (OpenTelemetry Automatic)
- `http_server_request_duration_seconds` - Request timing
- `http_server_active_requests` - Current request count
- `kestrel_active_connections` - ASP.NET Core connections
- `process_cpu_usage` - CPU utilization
- `process_memory_usage` - Memory consumption

## Known Issues / Limitations

1. **Metric Name Suffixes:**
   - OpenTelemetry adds unit suffix to metric names
   - Example: `alerts_received` → `alerts_received_alerts_total`
   - Dashboards need to use the actual exported names

2. **Dashboard Population:**
   - Grafana dashboards exist but query incorrect metric names
   - Need to update dashboard JSON with actual metric names:
     - `alerts_received_total` → `alerts_received_alerts_total`
     - `queue_depth` → `queue_depth_alerts`

3. **Missing Instrumentation (Future):**
   - Take-profit target hits (requires fill event processing)
   - Detailed API error tracking per endpoint
   - Order cancellation tracking with reasons

## Validation Commands

```bash
# Check Prometheus targets
curl -s 'http://localhost:9090/api/v1/targets' | jq '.data.activeTargets[] | {job: .labels.job, health: .health}'

# List available metrics
curl -s 'http://localhost:9090/api/v1/label/__name__/values' | jq -r '.data[]' | grep alerts

# Query specific metrics
curl -s 'http://localhost:9090/api/v1/query?query=alerts_received_alerts_total' | jq '.data.result'
curl -s 'http://localhost:9090/api/v1/query?query=alerts_processed_alerts_total' | jq '.data.result'
curl -s 'http://localhost:9090/api/v1/query?query=queue_depth_alerts' | jq '.data.result'
curl -s 'http://localhost:9090/api/v1/query?query=active_trades' | jq '.data.result'

# Check worker metrics directly
curl -s http://localhost:9464/metrics | grep "^alerts_"
```
