# Kill Switch Operations Guide

## Overview
The kill switch provides emergency controls to halt trading operations when needed. It offers three levels of intervention with different scopes of impact.

## Architecture

### Database Tables
- **system_state**: Key-value store for global system flags
  - `key = 'kill_switch'`
  - `value`: JSON containing `{active, level, reason, activated_at}`
- **kill_switch_audit**: Audit trail of all kill switch operations
  - Tracks: action (ACTIVATED/DEACTIVATED), level, reason, operator, timestamp, IP address

### Kill Switch Levels

#### PAUSE_NEW
- **Scope**: Stops alert processing only
- **Impact**: 
  - AlertWorker will not process new alerts from Redis queue
  - Existing trades continue to be monitored
  - Reconciliation continues running
- **Use Case**: Prevent new trades while investigating market conditions

#### PAUSE_ALL
- **Scope**: Pauses all background workers
- **Impact**:
  - AlertWorker stops processing alerts
  - ReconciliationWorker stops checking order consistency
  - TradeMonitorWorker stops monitoring open positions
  - Health checks continue (workers remain responsive)
- **Use Case**: Temporarily halt all automated operations

#### EMERGENCY_STOP
- **Scope**: Cancel all open orders + pause all operations
- **Impact**:
  - Immediately cancels ALL open orders on the exchange
  - Pauses all background workers (same as PAUSE_ALL)
  - Continues health checks
- **Use Case**: Emergency market conditions, suspected bugs, or immediate halt required

### Service Implementation
- **Caching**: Status queries cached for 5 seconds (reduces database load)
- **Emergency Cancellation**: Fetches all open orders and cancels each one sequentially
- **Persistence**: State stored in PostgreSQL for durability across restarts
- **Audit**: All activation/deactivation events logged with operator and reason

## API Reference

### Check Status
```bash
GET /api/killswitch/status
```

**Response:**
```json
{
  "active": false,
  "level": "PAUSE_ALL",
  "reason": null,
  "activatedAt": null
}
```

**Authentication**: None required (status is public)

### Activate Kill Switch
```bash
POST /api/killswitch/activate
Content-Type: application/json

{
  "secret": "your-secret-here",
  "level": "EMERGENCY_STOP",
  "reason": "Market conditions require immediate halt",
  "activatedBy": "operator-john"
}
```

**Parameters:**
- `secret`: Required. Must match `KillSwitchApi:Secret` configuration
- `level`: Required. One of: `PAUSE_NEW`, `PAUSE_ALL`, `EMERGENCY_STOP`
- `reason`: Required. Brief description of why kill switch is activated
- `activatedBy`: Required. Operator name/identifier for audit trail

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Kill switch activated at level EMERGENCY_STOP"
}
```

**Response (401 Unauthorized):**
```json
{
  "success": false,
  "message": "Invalid secret"
}
```

### Deactivate Kill Switch
```bash
POST /api/killswitch/deactivate
Content-Type: application/json

{
  "secret": "your-secret-here",
  "deactivatedBy": "operator-john",
  "reason": "Issue resolved, resuming operations"
}
```

**Parameters:**
- `secret`: Required. Must match `KillSwitchApi:Secret` configuration
- `deactivatedBy`: Required. Operator name/identifier for audit trail
- `reason`: Optional. Brief description of resolution

**Response (200 OK):**
```json
{
  "success": true,
  "message": "Kill switch deactivated"
}
```

## Emergency Procedures

### Scenario 1: Market Volatility - Prevent New Trades
**Situation**: Unusual market activity detected, want to stop new positions while monitoring existing ones.

**Action:**
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "level": "PAUSE_NEW",
    "reason": "High volatility detected, pausing new entries",
    "activatedBy": "monitoring-system"
  }'
```

**Verification:**
1. Check AlertWorker logs for "Kill switch active" messages
2. Verify ReconciliationWorker and TradeMonitorWorker still running
3. Monitor existing positions normally

**Resolution:**
Once market stabilizes:
```bash
curl -X POST http://localhost:8080/api/killswitch/deactivate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "deactivatedBy": "monitoring-system",
    "reason": "Market conditions normalized"
  }'
```

### Scenario 2: System Maintenance - Pause All Operations
**Situation**: Need to perform system maintenance or configuration changes.

**Action:**
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "level": "PAUSE_ALL",
    "reason": "System maintenance in progress",
    "activatedBy": "ops-team"
  }'
```

**During Maintenance:**
1. All workers paused but healthy (respond to health checks)
2. Existing orders remain on exchange (not cancelled)
3. Manual order management still possible via exchange UI

**Resolution:**
After maintenance:
```bash
curl -X POST http://localhost:8080/api/killswitch/deactivate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "deactivatedBy": "ops-team",
    "reason": "Maintenance completed successfully"
  }'
```

### Scenario 3: Emergency - Cancel Everything
**Situation**: Critical bug detected, exchange issues, or immediate halt required.

**Action:**
```bash
curl -X POST http://localhost:8080/api/killswitch/activate \
  -H "Content-Type: application/json" \
  -d '{
    "secret": "'$KILL_SWITCH_SECRET'",
    "level": "EMERGENCY_STOP",
    "reason": "Critical bug detected in position sizing",
    "activatedBy": "incident-responder"
  }'
```

**What Happens:**
1. All open orders cancelled immediately (one-by-one via exchange API)
2. All workers paused
3. System in safe state with no active orders

**Post-Emergency:**
1. Investigate root cause
2. Review audit trail:
   ```sql
   SELECT * FROM kill_switch_audit 
   WHERE action = 'ACTIVATED' AND level = 'EMERGENCY_STOP'
   ORDER BY ts DESC LIMIT 1;
   ```
3. Check cancelled orders:
   ```sql
   SELECT * FROM order_receipt 
   WHERE status = 'CANCELLED' 
     AND updated_at > (NOW() - INTERVAL '10 minutes')
   ORDER BY updated_at DESC;
   ```
4. Fix issue before reactivating system
5. Deactivate kill switch only when safe:
   ```bash
   curl -X POST http://localhost:8080/api/killswitch/deactivate \
     -H "Content-Type: application/json" \
     -d '{
       "secret": "'$KILL_SWITCH_SECRET'",
       "deactivatedBy": "incident-responder",
       "reason": "Bug fixed, system verified safe"
     }'
   ```

## Monitoring & Alerting

### Health Checks
Workers continue responding to health checks even when paused:
```bash
curl http://localhost:8080/health
curl http://localhost:8080/health/dependencies
```

### Database Monitoring
```sql
-- Current status
SELECT * FROM system_state WHERE key = 'kill_switch';

-- Recent activity
SELECT 
  action,
  level,
  reason,
  activated_by,
  ts,
  ip_address
FROM kill_switch_audit
ORDER BY ts DESC
LIMIT 20;

-- Time in activated state
SELECT 
  value->>'activated_at' as activated_at,
  EXTRACT(EPOCH FROM (NOW() - (value->>'activated_at')::timestamptz)) / 60 as minutes_active
FROM system_state 
WHERE key = 'kill_switch' 
  AND (value->>'active')::boolean = true;
```

### Log Messages
Workers log kill switch status:
```
[AlertWorker] Kill switch active at level PAUSE_NEW - skipping alert processing
[ReconciliationWorker] Kill switch active at level PAUSE_ALL - paused
[TradeMonitorWorker] Kill switch active at level EMERGENCY_STOP - paused
```

### Recommended Alerts
1. **Kill Switch Activated**: Alert operations team when activated
2. **Long Activation Duration**: Alert if active for > 60 minutes
3. **Unauthorized Activation Attempts**: Monitor failed API calls with invalid secrets

## Configuration

### Environment Variables
```bash
# Required for activate/deactivate endpoints
export KILL_SWITCH_SECRET="your-secure-secret-here"

# Or in appsettings.json / docker-compose.yml
KillSwitchApi__Secret=your-secure-secret-here
```

### Secret Management Best Practices
1. Use a strong, randomly generated secret (minimum 32 characters)
2. Store in environment variables, not code
3. Rotate periodically (e.g., quarterly)
4. Use different secrets per environment (dev/staging/prod)
5. Never commit secrets to version control

### Docker Compose Example
```yaml
environment:
  - KillSwitchApi__Secret=${KILL_SWITCH_SECRET}
```

In `.env` (not committed):
```bash
KILL_SWITCH_SECRET=your-secure-production-secret-here
```

## Testing

### Unit Tests
The kill switch service has unit tests covering:
- Initial inactive state
- Activation at each level
- Deactivation
- Emergency order cancellation behavior
- Status caching

Note: Integration tests requiring database are skipped by default. To run:
1. Start PostgreSQL: `docker compose up -d postgres`
2. Run init script: `psql -f scripts/db/init.sql`
3. Enable tests by removing `[Fact(Skip = "Requires database")]`

### Manual Testing Checklist
1. ✅ Activate PAUSE_NEW → verify AlertWorker paused, others running
2. ✅ Activate PAUSE_ALL → verify all workers paused
3. ✅ Activate EMERGENCY_STOP → verify orders cancelled + workers paused
4. ✅ Deactivate → verify workers resume processing
5. ✅ Invalid secret → verify 401 Unauthorized response
6. ✅ Status endpoint → verify public access (no auth)
7. ✅ Audit trail → verify all actions logged with operator details

## Troubleshooting

### Kill Switch Won't Activate
1. Check secret configuration: `echo $KILL_SWITCH_SECRET`
2. Verify API is running: `curl http://localhost:8080/health`
3. Check logs for authentication errors
4. Verify request format (JSON content-type, correct fields)

### Workers Not Pausing
1. Verify kill switch is active: `curl http://localhost:8080/api/killswitch/status`
2. Check worker logs for "Kill switch active" messages
3. Verify worker has IKillSwitchService injected (check DI registration)
4. Check database connection (workers need access to read status)

### Orders Not Cancelled on EMERGENCY_STOP
1. Check logs for "Cancelling all open orders" message
2. Verify ITradingProvider.GetOpenOrdersAsync() working
3. Check exchange API connectivity
4. Review individual order cancellation errors in logs
5. Manually verify order state on exchange UI

### Cache Issues
Status queries cached for 5 seconds. If stale data suspected:
1. Wait 6+ seconds and re-query
2. Check logs for cache misses
3. Restart API if cache corruption suspected (rare)

## Security Considerations

1. **Secret Protection**: Never expose kill switch secret in logs, error messages, or client responses
2. **Audit Trail**: All activation attempts (successful or failed) logged with IP address
3. **Rate Limiting**: Consider adding rate limiting to activation endpoints (future enhancement)
4. **Access Control**: Only operations team should have access to secret
5. **Network Security**: Consider restricting activation endpoints to internal network only
