# M7.1 Reconciliation Operations Guide

## Overview
The reconciliation system continuously monitors consistency between internal execution state and exchange order state, detecting discrepancies every 60 seconds.

## Discrepancy Types

### 1. MISSING_ON_EXCHANGE
**Cause:** Order was placed internally but doesn't exist on the exchange  
**Possible Reasons:**
- Order submission failed silently
- Exchange rejected the order after initial acceptance
- Network issues during order placement
- Exchange API downtime

**Resolution:**
```sql
-- Check the order receipt details
SELECT * FROM order_receipt 
WHERE execution_id = '<execution_id>' 
  AND exchange_order_id = '<missing_order_id>';

-- Check execution intent status
SELECT * FROM execution_intent WHERE execution_id = '<execution_id>';
```

**Actions:**
1. Verify order doesn't exist on exchange UI/API
2. Check execution logs for submission errors
3. Update execution_intent status to `FAILED` if order truly failed
4. Re-submit order if appropriate (consider market conditions)
5. Investigate API connectivity issues if pattern persists

### 2. ORPHANED_ON_EXCHANGE
**Cause:** Order exists on exchange but not tracked in internal system  
**Possible Reasons:**
- Manual order placement outside system
- System crash/restart lost tracking data
- Database rollback after order placement
- Order placed by another system/trader

**Resolution:**
```sql
-- Check if any execution matches this order
SELECT * FROM order_receipt 
WHERE exchange_order_id = '<orphaned_order_id>';

-- Check recent execution intents
SELECT * FROM execution_intent 
WHERE created_at > NOW() - INTERVAL '1 hour'
ORDER BY created_at DESC;
```

**Actions:**
1. Identify order on exchange (symbol, side, quantity, price)
2. Determine if order is legitimate system order or external
3. If external: Cancel on exchange immediately (security risk)
4. If legitimate but untracked: Create execution_intent and order_receipt retroactively
5. Review deployment/restart logs if untracked orders appear frequently

### 3. STATUS_MISMATCH
**Cause:** Order status differs between internal records and exchange  
**Possible Reasons:**
- Delayed status update notifications
- Webhook/WebSocket message loss
- Partial fill not properly recorded
- Exchange updated status without notification

**Resolution:**
```sql
-- Compare internal vs exchange status
SELECT 
    er.exchange_order_id,
    er.status as internal_status,
    rd.actual_state as exchange_state
FROM order_receipt er
JOIN reconciliation_discrepancy rd ON rd.execution_id = er.execution_id
WHERE rd.type = 'STATUS_MISMATCH'
  AND rd.detected_at > NOW() - INTERVAL '10 minutes';
```

**Actions:**
1. Fetch current order status from exchange API
2. Determine authoritative status (exchange is source of truth)
3. Update order_receipt.status to match exchange
4. Update execution_intent.status if needed
5. Investigate WebSocket connection stability if frequent mismatches

### 4. FILL_MISMATCH
**Cause:** Fill quantity or price differs between internal and exchange records  
**Possible Reasons:**
- Partial fill notification missed
- Fill price updated after initial capture
- Multiple fills not aggregated correctly
- Exchange fee calculation differences

**Resolution:**
```sql
-- Check fill details
SELECT 
    exchange_order_id,
    filled_quantity,
    average_fill_price,
    status
FROM order_receipt
WHERE execution_id = '<execution_id>';
```

**Actions:**
1. Query exchange for complete fill history
2. Calculate correct aggregated fills (quantity, average price)
3. Update order_receipt with accurate fill data
4. Verify fee calculations match exchange
5. Review fill notification handling if pattern emerges

### 5. INVALIDATION_TRIGGERED
**Cause:** Stop-loss or take-profit hit but status not updated  
**Possible Reasons:**
- Market moved through stop price rapidly
- Invalidation check lag vs exchange execution
- Stop order triggered but notification delayed
- Exchange rejected stop order activation

**Resolution:**
```sql
-- Check invalidation configuration
SELECT 
    ei.execution_id,
    ei.invalidation_price,
    ei.status,
    rd.actual_state
FROM execution_intent ei
JOIN reconciliation_discrepancy rd ON rd.execution_id = ei.execution_id
WHERE rd.type = 'INVALIDATION_TRIGGERED';
```

**Actions:**
1. Verify stop order filled on exchange
2. Update execution_intent.status to `INVALIDATED`
3. Ensure no position remains open
4. Review stop loss triggering logic if false positives occur
5. Check market data feed latency

### 6. ERROR
**Cause:** Reconciliation process encountered an error  
**Possible Reasons:**
- Exchange API timeout
- Rate limit exceeded
- Network connectivity issues
- Malformed order data

**Resolution:**
```sql
-- Check error details
SELECT 
    execution_id,
    expected_state,
    actual_state,
    details
FROM reconciliation_discrepancy
WHERE type = 'ERROR'
  AND detected_at > NOW() - INTERVAL '30 minutes';
```

**Actions:**
1. Review error message in `details` column
2. Check exchange API status page
3. Verify rate limits not exceeded
4. Test exchange connectivity
5. Investigate data integrity if errors persist

## Monitoring Queries

### Recent Discrepancies Summary
```sql
SELECT 
    type,
    COUNT(*) as count,
    MAX(detected_at) as latest
FROM reconciliation_discrepancy
WHERE detected_at > NOW() - INTERVAL '1 hour'
GROUP BY type
ORDER BY count DESC;
```

### Reconciliation Health
```sql
SELECT 
    checked_at,
    executions_checked,
    discrepancies_found,
    reconciliation_type
FROM reconciliation_state
ORDER BY checked_at DESC
LIMIT 20;
```

### Unresolved Discrepancies
```sql
-- Discrepancies that keep appearing (not resolved)
SELECT 
    execution_id,
    type,
    COUNT(*) as occurrence_count,
    MIN(detected_at) as first_seen,
    MAX(detected_at) as last_seen
FROM reconciliation_discrepancy
WHERE detected_at > NOW() - INTERVAL '1 day'
GROUP BY execution_id, type
HAVING COUNT(*) > 3
ORDER BY occurrence_count DESC;
```

### Active Executions State
```sql
SELECT 
    ei.execution_id,
    ei.symbol,
    ei.side,
    ei.status as intent_status,
    COUNT(er.receipt_id) as order_count,
    SUM(CASE WHEN er.status = 'FILLED' THEN 1 ELSE 0 END) as filled_orders
FROM execution_intent ei
LEFT JOIN order_receipt er ON er.execution_id = ei.execution_id
WHERE ei.status IN ('PENDING', 'PARTIAL', 'ACTIVE')
GROUP BY ei.execution_id, ei.symbol, ei.side, ei.status;
```

## Configuration

### Reconciliation Interval
Default: 60 seconds

To adjust polling frequency, update `appsettings.json`:
```json
{
  "Reconciliation": {
    "IntervalSeconds": 60
  }
}
```

**Considerations:**
- Lower intervals (30s) = faster detection, higher API load
- Higher intervals (120s+) = lower API costs, slower detection
- Exchange rate limits must be considered

### Alerting (Future Enhancement)
Currently, discrepancies are logged and persisted to database. Future enhancements could include:
- Email/SMS alerts for critical discrepancies
- PagerDuty integration for ORPHANED_ON_EXCHANGE
- Slack notifications for operator review
- Automated recovery actions for common scenarios

## Best Practices

1. **Review Daily:** Check reconciliation_discrepancy table daily for patterns
2. **Investigate Orphans Immediately:** Orphaned orders are potential security risks
3. **Trust Exchange State:** When in doubt, exchange is source of truth
4. **Document Resolution:** Note resolution actions in operations log
5. **Monitor Trends:** Increasing discrepancies may indicate infrastructure issues
6. **Test Connectivity:** Periodic exchange API health checks
7. **Audit Logs:** Correlate discrepancies with deployment/restart events

## Emergency Response

### Critical: Unauthorized Orders Detected
```bash
# Immediately cancel all orders on exchange
curl -X POST http://localhost:8080/admin/cancel-all-orders \
  -H "Authorization: Bearer <admin-token>"

# Then investigate source in database
```

### High: Multiple Missing Orders
```bash
# Check exchange connectivity
curl https://demo-futures.kraken.com/derivatives/api/v3/instruments

# Review recent deployments/restarts
docker logs trading-api | grep -i "reconciliation\|error"

# Verify database connectivity
psql -h localhost -U postgres -d ai-trading-db -c "SELECT COUNT(*) FROM execution_intent WHERE status = 'PENDING';"
```
