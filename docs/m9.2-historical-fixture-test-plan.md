## Critical Understanding: Validation Data vs. Test Data

### Validation Data (Historical Lookback for Analysis)
**Purpose:** Provide sufficient candles for indicator calculation and Elliott Wave pivot extraction

**Per-Timeframe Requirements:**

| Timeframe | Lookback | Bars Required | Days of Data | Used For |
|-----------|----------|---------------|--------------|----------|
| **M5** | 1 day | 288 candles | ~1.5 days | RSI, MACD, StochRSI calculation |
| **M15** | 1 day | 96 candles | ~1.5 days | Multi-timeframe indicator analysis |
| **M30** | 1 day | 48 candles | ~1.5 days | Pattern confirmation |
| **H1** | 2 days | 48 candles | ~2.5 days | Trend analysis |
| **H2** | 3 days | 36 candles | ~3.5 days | Macro trend context |

**Elliott Wave Requirements:**
- **Minimum lookback:** 800 bars (per M4 low-level spec)
- **Target:** ~1,200 bars for reliable pivot extraction (12+ pivots minimum)
- **Base timeframe:** Always use base timeframe from alert (e.g., M5, M15)

**⚠️ Important:** We need **MORE than the lookback period** to ensure we have enough historical data:
- For M5: Need **~4 days** of data to safely provide 1.5 days lookback
- For H1: Need **~7 days** of data to safely provide 2.5 days lookback
- For H2: Need **~10 days** of data to safely provide 3.5 days lookback

**📊 Data Collection Strategy:** Collect **30 days** of data to enable testing any 4-day window in the past month. This provides:
- Sufficient lookback for H2 timeframe (3.5 days) + Elliott (800-1200 bars)
- Flexibility to select different test periods without re-fetching data
- Ability to test multiple non-overlapping 4-day windows

### Test Data (Entry Trigger Point)
**Purpose:** The specific candle where we simulate an alert and trigger trade decision

**Characteristics:**
- Single candle timestamp (e.g., `2026-01-08T05:15:00Z`)
- Must be AFTER all validation data in the fixture
- Represents "current price action" that triggers the alert
- Direction (LONG/SHORT) based on Pine Script alert signal

**Example:**
```
Historical Data: [4 days of M5 candles] ← Validation Data (analysis)
                                    ↓
Test Point: 2026-01-08T05:15:00Z ← Test Data (entry trigger)
```

## Generic & Reusable Test Design

### Why 30 Days of Data?

This test plan is designed to be **generic and reusable** for testing different historical periods **and trading sessions** without re-fetching data:

1. **One-Time Setup:** Fetch 30 days of OHLCV data once
2. **Multiple Test Windows:** Select any 4-day window within the 30 days
3. **Multiple Trading Sessions:** Test morning (04:30-07:30), US open (14:00-16:00), Asian (00:00-03:00), or any custom time range
4. **Market Condition Variety:** Test volatile periods, range-bound periods, trending periods
5. **Sufficient Lookback:** Even the earliest test points (day 5-8) have 4+ days of prior data for lookback
6. **No Re-fetching:** Change test window and time-of-day by updating parameters only

## Pipeline Dataflow (What We're Testing)

```
┌──────────────────┐
│ Fake Alert       │  Symbol: BTCUSD.P, Direction: LONG/SHORT
│ (HTTP POST)      │  Timestamp: Historical (e.g., 2026-01-08T05:15:00Z)
└────────┬─────────┘
         ↓
┌──────────────────┐
│ API Webhook      │  Validates secret, enqueues to Redis
│ Endpoint         │  Returns correlation_id
└────────┬─────────┘
         ↓
┌──────────────────┐
│ AlertWorker      │  Dequeues from Redis, starts processing
│ (Background)     │  Symbol normalized: BTCUSD.P → PI_XBTUSD
└────────┬─────────┘
         ↓
┌──────────────────┐
│ Market Data      │  Mode: fixtures (not live API!)
│ Fetch            │  Loads from: tests/fixtures/kraken-futures/pf_xbtusd_m{interval}.json
└────────┬─────────┘
         ↓
┌──────────────────┐
│ Indicator        │  Calculates RSI, Stoch RSI, MACD, Volume
│ Snapshot         │  Multi-timeframe: M5, M15, M30, H1, H2
└────────┬─────────┘  Output: IndicatorSnapshot with risk category
         ↓
┌──────────────────┐
│ Elliott Engine   │  Extracts pivots via ZigZag
│                  │  Builds impulse candidates with rule checks
└────────┬─────────┘  Output: ElliottCandidates (5+ pivots required)
         ↓
┌──────────────────┐
│ LLM Adjudication │  MCP gateway invokes LLM (local or OpenAI)
│                  │  Decision: ALLOWLONGW3, ALLOWLONGW5END, ALLOWSHORTW3,
└────────┬─────────┘           ALLOWSHORTW5END, REJECT, UNCERTAIN
         ↓
┌──────────────────┐
│ Risk Engine      │  If ALLOW: calculate position size, stop loss
│                  │  Applies risk policy (daily limits, equity %)
└────────┬─────────┘  Output: TradePlan with 3+ partial targets
         ↓
┌──────────────────┐
│ Execution        │  Demo mode: no real orders
│ Service          │  Persists execution intent + order receipt
└────────┬─────────┘  Records in database for audit trail
         ↓
┌──────────────────┐
│ Database         │  17 tables updated:
│ Persistence      │  - alerts, alert_processing, indicator_snapshots
│                  │  - elliott_candidates, llm_adjudications, trade_plan
└──────────────────┘  - execution_intent, order_receipt, etc.
```

### Step 2.1: Create Helper Script (extract-volatile-candles.sh)

Create `scripts/fixtures/extract-volatile-candles.sh`:

```bash
#!/usr/bin/env bash
# Extract candles with significant price movements from a specified date and time range
# FULLY GENERIC: Works for any date range and any time-of-day window

set -euo pipefail

FIXTURE_FILE="${1:-tests/fixtures/kraken-futures/pf_xbtusd_m5.json}"
MIN_MOVE_PCT="${2:-1.0}"       # Minimum price movement % (default: 1%)
START_DATE="${3:-}"            # ISO 8601: 2026-01-08T00:00:00Z (optional)
END_DATE="${4:-}"              # ISO 8601: 2026-01-12T23:59:59Z (optional)
START_TIME="${5:-04:30}"       # Time of day start (default: 04:30 UTC - morning session)
END_TIME="${6:-07:30}"         # Time of day end (default: 07:30 UTC - morning session)

# If dates not provided, use last 4 days of dataset
if [[ -z "$START_DATE" ]]; then
  # Calculate dates dynamically from fixture file
  LAST_TIMESTAMP=$(jq -r '.candles[-1].time' "$FIXTURE_FILE")
  START_DATE=$(date -u -d "$LAST_TIMESTAMP - 4 days" +%Y-%m-%dT00:00:00Z 2>/dev/null || \
               date -u -v-4d -j -f "%Y-%m-%dT%H:%M:%SZ" "$LAST_TIMESTAMP" +%Y-%m-%dT00:00:00Z)
  END_DATE="$LAST_TIMESTAMP"
fi

jq -r --arg min_pct "$MIN_MOVE_PCT" --arg start_date "$START_DATE" --arg end_date "$END_DATE" \
      --arg start_time "$START_TIME" --arg end_time "$END_TIME" '
  .candles[] | 
  select(
    # Filter by date range
    (.time >= $start_date and .time <= $end_date)
  ) |
  select(
    # Filter by time of day (flexible - uses START_TIME and END_TIME variables)
    (.time | fromdateiso8601 | strftime("%H:%M") | 
     (. >= $start_time and . <= $end_time))
  ) |
  select(
    # Filter by minimum price movement
    (((.high - .low) / .low * 100) >= ($min_pct | tonumber))
  ) |
  [.time, .open, .high, .low, .close, 
   ((.high - .low) / .low * 100 | tostring + "%")] |
  @tsv
' "$FIXTURE_FILE"
```
