# M7.3 Metrics & Observability Guide

## Overview
This guide provides comprehensive documentation for the metrics, health checks, and observability features of the MVP Trading system.

## Metrics Architecture

### Technology Stack
- **Metrics SDK**: OpenTelemetry 1.9.0
- **Exporter (API)**: Prometheus AspNetCore exporter (port 8080)
- **Exporter (Worker)**: Prometheus HttpListener exporter (port 9464)
- **Backend**: Prometheus (latest)
- **Visualization**: Grafana (latest)

### Metrics Service
The `IMetricsService` interface provides a unified API for recording metrics across all components:
- Located in `Mvp.Trading.Contracts.Telemetry`
- Implemented by `OpenTelemetryMetricsService`
- Registered as singleton in both API and Worker

## Metrics Catalog

### Alert Processing Metrics

#### `alerts_received_total` (Counter)
Total number of alerts received from TradingView webhooks.

**Labels:**
- `exchange` - Exchange identifier (e.g., "krakenfutures")
- `symbol` - Trading symbol (e.g., "BTCUSD.P")

**Prometheus Queries:**
```promql
# Alert rate per second (1-minute window)
rate(alerts_received_total[1m])

# Alerts by symbol
sum by (symbol) (alerts_received_total)

# Alerts by exchange
sum by (exchange) (alerts_received_total)
```

#### `alerts_processed_total` (Counter)
Total number of alerts processed with their outcomes.

**Labels:**
- `outcome` - Processing result:
  - `accepted` - Alert successfully processed and executed
  - `rejected_indicator` - Rejected during indicator computation
  - `rejected_elliott` - Rejected due to Elliott wave analysis
  - `rejected_llm` - Rejected by LLM adjudication
  - `error` - Processing failed with error

**Prometheus Queries:**
```promql
# Success rate (accepted / total)
sum(rate(alerts_processed_total{outcome="accepted"}[5m])) / sum(rate(alerts_processed_total[5m]))

# Rejection breakdown
sum by (outcome) (rate(alerts_processed_total{outcome=~"rejected.*"}[5m]))

# Error rate
rate(alerts_processed_total{outcome="error"}[5m])
```

#### `alert_processing_duration_seconds` (Histogram)
Duration of alert processing from receipt to execution/rejection.

**Buckets**: Auto-generated by OpenTelemetry

**Prometheus Queries:**
```promql
# 95th percentile processing time
histogram_quantile(0.95, rate(alert_processing_duration_seconds_bucket[5m]))

# 99th percentile processing time
histogram_quantile(0.99, rate(alert_processing_duration_seconds_bucket[5m]))

# Average processing time
rate(alert_processing_duration_seconds_sum[5m]) / rate(alert_processing_duration_seconds_count[5m])
```

#### `queue_depth` (Gauge)
Current number of alerts waiting in Redis queue.

**Prometheus Queries:**
```promql
# Current queue depth
queue_depth

# Max queue depth over last hour
max_over_time(queue_depth[1h])

# Alert if queue depth exceeds 100
queue_depth > 100
```

### Execution Metrics

#### `executions_processed_total` (Counter)
Total number of execution requests processed.

**Labels:**
- `outcome` - Execution result:
  - `filled` - Order filled immediately
  - `placed` - Order placed successfully
  - `rejected_risk` - Rejected by risk management
  - `rejected_heartbeat` - Rejected due to missing heartbeat
  - `error` - Execution failed

**Prometheus Queries:**
```promql
# Execution success rate
sum(rate(executions_processed_total{outcome=~"filled|placed"}[5m])) / sum(rate(executions_processed_total[5m]))

# Risk rejection rate
rate(executions_processed_total{outcome="rejected_risk"}[5m])
```

#### `execution_duration_seconds` (Histogram)
Duration of execution stages.

**Labels:**
- `stage` - Execution phase:
  - `total` - End-to-end execution time
  - `llm_adjudication` - LLM decision time
  - `order_placement` - Order submission time

**Prometheus Queries:**
```promql
# 95th percentile total execution time
histogram_quantile(0.95, rate(execution_duration_seconds_bucket{stage="total"}[5m]))

# LLM adjudication latency
histogram_quantile(0.99, rate(execution_duration_seconds_bucket{stage="llm_adjudication"}[5m]))
```

#### `orders_placed_total` (Counter)
Total number of orders placed on the exchange.

**Labels:**
- `direction` - Trade direction: `LONG` or `SHORT`
- `order_type` - Order type: `LIMIT` or `MARKET`

**Prometheus Queries:**
```promql
# Orders by direction
sum by (direction) (orders_placed_total)

# Limit vs Market order ratio
sum(orders_placed_total{order_type="LIMIT"}) / sum(orders_placed_total)
```

#### `orders_cancelled_total` (Counter)
Total number of cancelled orders.

**Labels:**
- `reason` - Cancellation reason (e.g., "stop_loss", "take_profit", "manual")

**Prometheus Queries:**
```promql
# Cancellation rate by reason
sum by (reason) (rate(orders_cancelled_total[5m]))
```

#### `orders_filled_total` (Counter)
Total number of filled orders.

**Labels:**
- `symbol` - Trading symbol
- `direction` - Trade direction

**Prometheus Queries:**
```promql
# Fill rate per symbol
sum by (symbol) (rate(orders_filled_total[5m]))

# Long vs Short fills
sum by (direction) (orders_filled_total)
```

#### `active_trades` (Gauge)
Current number of open positions.

**Prometheus Queries:**
```promql
# Current open positions
active_trades

# Alert if too many open positions
active_trades > 10
```

### Risk Management Metrics

#### `stop_loss_triggered_total` (Counter)
Total number of stop-loss triggers.

**Labels:**
- `symbol` - Trading symbol

**Prometheus Queries:**
```promql
# Stop-loss trigger rate
sum(rate(stop_loss_triggered_total[5m]))

# Most stopped-out symbols
topk(5, sum by (symbol) (stop_loss_triggered_total))
```

#### `take_profit_hit_total` (Counter)
Total number of take-profit hits.

**Labels:**
- `symbol` - Trading symbol
- `target` - Target number (1, 2, 3, etc.)

**Prometheus Queries:**
```promql
# Take-profit hit rate
sum(rate(take_profit_hit_total[5m]))

# Hits by target level
sum by (target) (take_profit_hit_total)
```

### Error Metrics

#### `errors_total` (Counter)
Total number of errors by component.

**Labels:**
- `component` - Source component (e.g., "AlertWorker", "ExecutionService")
- `error_type` - Exception type (e.g., "InvalidOperationException")

**Prometheus Queries:**
```promql
# Error rate by component
sum by (component) (rate(errors_total[5m]))

# Most common errors
topk(10, sum by (error_type) (errors_total))
```

#### `api_errors_total` (Counter)
Total number of API errors (external service calls).

**Labels:**
- `provider` - API provider (e.g., "kraken", "openai")
- `endpoint` - API endpoint
- `error_code` - HTTP status code or error code

**Prometheus Queries:**
```promql
# API error rate by provider
sum by (provider) (rate(api_errors_total[5m]))

# 5xx errors
sum(rate(api_errors_total{error_code=~"5.."}[5m]))
```

### Reconciliation Metrics

#### `reconciliation_discrepancies` (Gauge)
Current number of unresolved reconciliation discrepancies.

**Prometheus Queries:**
```promql
# Current discrepancies
reconciliation_discrepancies

# Alert if discrepancies detected
reconciliation_discrepancies > 0
```

## Health Check Endpoints

### `/health` - Basic Health
Simple liveness check that returns 200 OK if the API process is running.

**Response:**
```json
{
  "status": "ok"
}
```

### `/health/dependencies` - Dependency Health
Checks connectivity to Postgres and Redis.

**Response (Healthy):**
```json
{
  "status": "Healthy",
  "totalDuration": "00:00:00.0234567",
  "entries": {
    "postgres": {
      "status": "Healthy"
    },
    "redis": {
      "status": "Healthy"
    }
  }
}
```

**Response (Unhealthy):**
Returns 503 Service Unavailable with error details.

### `/health/ready` - Readiness Probe
Kubernetes readiness probe - checks if service can accept traffic.

### `/health/live` - Liveness Probe
Kubernetes liveness probe - checks if service is alive and shouldn't be restarted.

### Alerting Rules

#### High Error Rate
```yaml
- alert: HighErrorRate
  expr: sum(rate(errors_total[5m])) > 1
  for: 5m
  annotations:
    summary: "High error rate detected"
    description: "Error rate is {{ $value }} errors/sec"
```

#### Queue Depth Alert
```yaml
- alert: HighQueueDepth
  expr: queue_depth > 100
  for: 10m
  annotations:
    summary: "Alert queue depth is high"
    description: "Queue depth is {{ $value }} alerts"
```

#### Reconciliation Discrepancy
```yaml
- alert: ReconciliationDiscrepancy
  expr: reconciliation_discrepancies > 0
  for: 15m
  annotations:
    summary: "Reconciliation discrepancies detected"
    description: "{{ $value }} unresolved discrepancies"
```

## Common Prometheus Queries

### System Overview
```promql
# Total alerts processed in last 24h
sum(increase(alerts_processed_total[24h]))

# Success rate (last hour)
sum(rate(alerts_processed_total{outcome="accepted"}[1h])) / sum(rate(alerts_processed_total[1h]))

# Average processing time (last 5 minutes)
rate(alert_processing_duration_seconds_sum[5m]) / rate(alert_processing_duration_seconds_count[5m])
```

### Performance Analysis
```promql
# Top 5 slowest endpoints
topk(5, histogram_quantile(0.99, rate(execution_duration_seconds_bucket{stage="total"}[5m])))

# Processing time by outcome
avg by (outcome) (rate(alert_processing_duration_seconds_sum[5m]) / rate(alert_processing_duration_seconds_count[5m]))
```

### Error Investigation
```promql
# Errors in last hour
sum(increase(errors_total[1h]))

# Error rate trend (5-minute windows)
sum(rate(errors_total[5m]))

# Most common error types
topk(10, sum by (error_type) (errors_total))
```

## Troubleshooting

### Metrics Not Appearing

**Problem**: Metrics not visible in Prometheus

**Solutions:**
1. Check scrape targets in Prometheus UI: http://localhost:9090/targets
2. Verify metrics endpoints are accessible:
   ```bash
   curl http://localhost:8080/metrics
   curl http://localhost:9464/metrics
   ```
3. Check Prometheus logs:
   ```bash
   docker compose logs prometheus
   ```
4. Verify network connectivity between containers

### Grafana Can't Connect to Prometheus

**Problem**: Grafana shows "Bad Gateway" or connection errors

**Solutions:**
1. Verify Prometheus is running:
   ```bash
   docker compose ps prometheus
   ```
2. Check datasource configuration in Grafana UI
3. Test connection from Grafana container:
   ```bash
   docker compose exec grafana curl http://prometheus:9090
   ```

### High Cardinality Issues

**Problem**: Too many unique label combinations causing memory issues

**Solutions:**
1. Limit label values (avoid unbounded strings like order IDs)
2. Use exemplars instead of labels for high-cardinality data
3. Adjust Prometheus retention settings
4. Consider aggregation at the application level

## Best Practices

### Metric Naming
- Use `_total` suffix for counters
- Use `_seconds` for durations
- Use descriptive label names
- Keep label cardinality low (< 100 unique values per label)

### Query Performance
- Use recording rules for frequently-used queries
- Avoid regex matchers when possible
- Use rate() instead of irate() for smoother graphs
- Limit time ranges for expensive queries

### Dashboard Design
- Group related metrics together
- Use consistent time ranges across panels
- Add annotations for deployments and incidents
- Include both current values and trends

### Alerting
- Set appropriate thresholds based on baseline metrics
- Use `for` duration to avoid alert fatigue
- Include actionable information in alert descriptions
- Test alerts in staging before production
